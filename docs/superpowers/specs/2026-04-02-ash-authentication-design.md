# Ash Authentication for Panko

## Problem

Panko currently uses an optional `PANKO_API_KEY` environment variable for access control. This requires passing the key as a query parameter or Bearer token, which is clunky for browser usage. The goal is to replace this with a proper username/password login flow so the main user can click around without auth friction, while still gating access to prevent unauthorized viewing of unshared sessions. Public share routes (`/s/:slug`) must remain unauthenticated.

## Decision

Use `ash_authentication` and `ash_authentication_phoenix` for a password-based auth system. This is the Ash-native approach with minimal custom code. All logged-in users see all sessions (no per-user scoping in this phase). Per-user scoping via Ash policies is deferred to a future phase — the `user_id` columns on sessions and shares are already in the schema for that.

## Scope

**In scope:**
- User resource with email/password authentication
- Token resource for session management
- Sign-up (registration) and sign-in flows
- Sign-out
- Protected LiveView routes (sessions, session detail)
- Public share routes stay unauthenticated
- Keep existing API key plug as fallback for API/programmatic access

**Out of scope:**
- Per-user session/share scoping (future: Ash policies)
- OAuth/SSO strategies (future: add to authentication block)
- Password reset email sending (stub the sender, no email config)
- Admin role or user management UI
- Email confirmation requirement

## Architecture

### New Domain: `Panko.Accounts`

```
lib/panko/accounts.ex           # Ash.Domain
lib/panko/accounts/user.ex      # User resource (AshAuthentication)
lib/panko/accounts/token.ex     # Token resource (AshAuthentication.TokenResource)
```

Register in config:
```elixir
config :panko, ash_domains: [Panko.Sessions, Panko.Sharing, Panko.Accounts]
```

### User Resource (`Panko.Accounts.User`)

```elixir
use Ash.Resource,
  domain: Panko.Accounts,
  data_layer: AshPostgres.DataLayer,
  extensions: [AshAuthentication]
```

Attributes:
- `id` — UUID primary key
- `email` — `:ci_string`, not null, public
- `hashed_password` — `:string`, not null, sensitive

Identities:
- `identity :unique_email, [:email]` — required by password strategy

Authentication block:
- `tokens` — enabled, `store_all_tokens? true`, backed by `Panko.Accounts.Token`, signing secret via function:
  ```elixir
  signing_secret fn _, _ ->
    Application.fetch_env(:panko, :token_signing_secret)
  end
  ```
- `strategies.password` — identity field `:email`, registration enabled, sign-in enabled
- No password reset sender (omit `resettable` block for now)

Actions:
- `defaults [:read]`
- `read :get_by_subject` — required by AshAuthentication, manually defined:
  ```elixir
  read :get_by_subject do
    description "Get a user by the subject claim in a JWT"
    argument :subject, :string, allow_nil?: false
    get? true
    prepare AshAuthentication.Preparations.FilterBySubject
  end
  ```
- `register_with_password` and `sign_in_with_password` — auto-generated by password strategy

Postgres: table `"users"`, repo `Panko.Repo`

### Token Resource (`Panko.Accounts.Token`)

Extensions: `AshAuthentication.TokenResource`, `AshPostgres.DataLayer`

Minimal resource — the extension auto-generates all attributes and actions.

Postgres: table `"tokens"`, repo `Panko.Repo`

### Accounts Domain (`Panko.Accounts`)

Resources: `User`, `Token`

### Signing Secret

Runtime config via `PANKO_TOKEN_SIGNING_SECRET` env var. Generate a default for dev/test. The secret must not be committed to source control.

```elixir
# config/dev.exs
config :panko, :token_signing_secret, "dev-only-secret-at-least-32-bytes-long!!"

# config/test.exs
config :panko, :token_signing_secret, "test-only-secret-at-least-32-bytes-long!!"

# config/runtime.exs
if config_env() == :prod do
  config :panko, :token_signing_secret,
    System.get_env("PANKO_TOKEN_SIGNING_SECRET") ||
      raise "PANKO_TOKEN_SIGNING_SECRET env var is required in production"
end
```

## Router Changes

### Add AshAuthentication.Phoenix.Router

Add to `router.ex` after the existing `use PankoWeb, :router`:

```elixir
use AshAuthentication.Phoenix.Router
import AshAuthentication.Plug.Helpers
```

This imports `load_from_session`, `load_from_bearer`, `store_in_session`, `auth_routes`, `sign_in_route`, `sign_out_route`, and the `ash_authentication_live_session` macro.

### Pipeline Changes

Update browser pipeline to load auth from session:

```elixir
pipeline :browser do
  plug :accepts, ["html"]
  plug :fetch_session
  plug :fetch_live_flash
  plug :put_root_layout, html: {PankoWeb.Layouts, :root}
  plug :protect_from_forgery
  plug :put_secure_browser_headers
  plug :load_from_session       # NEW: load authenticated user from session
end
```

### Route Structure

```elixir
# Public auth routes (sign in, register, sign out)
scope "/", PankoWeb do
  pipe_through :browser

  sign_in_route(register_path: "/register", auth_routes_prefix: "/auth")
  sign_out_route AuthController
  auth_routes AuthController, Panko.Accounts.User, path: "/auth"
end

# Protected routes — require authenticated user
# NOTE: scope and pipe_through MUST wrap ash_authentication_live_session,
# not be inside it. The macro wraps live_session which cannot contain pipe_through.
scope "/", PankoWeb do
  pipe_through :browser

  ash_authentication_live_session :authenticated,
    otp_app: :panko,
    on_mount_prepend: [{PankoWeb.LiveUserAuth, :live_user_required}],
    layout: {PankoWeb.Layouts, :app} do
    live "/", SessionsLive, :index
    live "/sessions/:id", SessionLive, :show
  end
end

# Public share routes — no auth
scope "/s", PankoWeb do
  pipe_through :browser

  live_session :public do
    live "/:slug", ShareLive, :show
  end
end
```

### Remove `maybe_require_api_key` Pipeline

The `PankoWeb.Plugs.ApiKeyAuth` plug is replaced by Ash Authentication's session-based auth for browser routes. Keep the plug file for potential future API pipeline use, but remove it from the browser pipeline.

## Auth Controller

New file: `lib/panko_web/controllers/auth_controller.ex`

Implements `AshAuthentication.Phoenix.Controller` behaviour:

```elixir
defmodule PankoWeb.AuthController do
  use PankoWeb, :controller
  use AshAuthentication.Phoenix.Controller

  def success(conn, _activity, user, _token) do
    conn
    |> store_in_session(user)
    |> assign(:current_user, user)
    |> redirect(to: ~p"/")
  end

  def failure(conn, _activity, _reason) do
    conn
    |> put_flash(:error, "Authentication failed")
    |> redirect(to: ~p"/sign-in")
  end

  def sign_out(conn, _params) do
    conn
    |> clear_session()
    |> redirect(to: ~p"/sign-in")
  end
end
```

## Migrations

Two new tables via `mix ash.codegen`:

### `users` table
- `id` — UUID primary key
- `email` — citext, not null, unique (requires `citext` extension — already enabled in repo)
- `hashed_password` — text, not null
- `inserted_at` / `updated_at` — timestamps

### `tokens` table
- Auto-generated by `AshAuthentication.TokenResource`
- Includes: `jti`, `subject`, `expires_at`, `purpose`, `extra_data`, `created_at`

## LiveView Changes

### Current User Access

In protected LiveViews, `@current_user` is automatically available via the `ash_authentication_live_session` on_mount hook. No manual assigns needed.

### Redirect Unauthenticated

Create `lib/panko_web/live_user_auth.ex` with an `on_mount` hook that redirects unauthenticated users to `/sign-in`:

```elixir
defmodule PankoWeb.LiveUserAuth do
  import Phoenix.LiveView
  import Phoenix.Component

  def on_mount(:live_user_required, _params, _session, socket) do
    if socket.assigns[:current_user] do
      {:cont, socket}
    else
      {:halt, redirect(socket, to: ~p"/sign-in")}
    end
  end
end
```

This is prepended to the `ash_authentication_live_session` via `on_mount_prepend` (see Router section).
